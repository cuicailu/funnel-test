# 前端测试标识（data-testid）规范

## 1. 规范目的

随着系统复杂度的提升，为了保障产品质量并提升回归测试效率，团队将逐步引入 UI 自动化测试。
当前前端页面元素多采用 CSS 类名（Class）进行定位，这种方式存在极大的脆弱性：**一旦前端重构样式或修改类名，自动化测试脚本就会大面积失效，导致维护成本极高。**

为解决此问题，本规范要求在所有核心交互元素上添加专属的测试属性 `data-testid`。该属性与业务逻辑和 UI 样式完全解耦，专供自动化测试脚本定位使用，从而实现测试用例的稳定运行，大幅降低后续的维护成本。

## 2. 适用范围

本规范适用于所有新开发的前端页面及组件。对于历史页面，将在后续迭代中逐步补充。

**必须添加 `data-testid` 的元素类型包括：**
- **所有可点击的交互元素**：按钮（Button）、链接（Link）、标签页（Tab）、图标（Icon）等。
- **所有表单输入元素**：输入框（Input）、下拉框（Select）、单选/复选框（Radio/Checkbox）、日期选择器（DatePicker）等。
- **核心数据展示容器**：表格行（Table Row）、列表项（List Item）、图表画布（Canvas/Chart Container）、弹窗容器（Modal/Dialog）等。
- **关键状态标识元素**：成功/失败提示（Toast/Message）、空状态提示（Empty State）等。

## 3. 命名规范

`data-testid` 的命名必须遵循以下结构，以确保全局唯一性且易于理解：

**格式：`[模块名]-[组件类型]-[功能描述]`**

### 3.1 命名规则要求

1. **全小写字母**：所有单词必须小写。
2. **中划线连接**：单词之间使用中划线 `-` 连接，严禁使用驼峰命名或下划线。
3. **语义化**：名称必须能清晰表达该元素在当前业务模块中的具体作用。

### 3.2 字段说明

| 字段 | 说明 | 示例 |
|---|---|---|
| **模块名** | 当前页面或业务模块的英文简写 | `funnel` (漏斗分析), `report` (报表), `contact` (联系人) |
| **组件类型** | 元素的 UI 类型简写 | `btn` (按钮), `input` (输入框), `select` (下拉框), `tab` (标签页), `list` (列表), `modal` (弹窗) |
| **功能描述** | 该元素的具体业务动作或含义 | `save` (保存), `delete` (删除), `search` (搜索), `submit` (提交) |

### 3.3 常见组件类型缩写对照表

| 组件类型 | 缩写 | 示例 |
|---|---|---|
| 按钮 | `btn` | `funnel-btn-save` |
| 输入框 | `input` | `contact-input-name` |
| 下拉选择框 | `select` | `report-select-type` |
| 标签页 | `tab` | `funnel-tab-filter` |
| 列表项/表格行 | `item` | `contact-item-row` |
| 弹窗/对话框 | `modal` | `report-modal-export` |
| 图标 | `icon` | `funnel-icon-chart-type` |
| 容器/区域 | `container` | `funnel-container-chart` |

## 4. 动态列表元素的特殊处理

对于列表、表格等多行数据，为了在测试中能精准定位到某一行，`data-testid` 需要结合动态数据（如 ID 或 Index）进行拼接。

**格式：`[模块名]-[组件类型]-[功能描述]-[唯一标识]`**

**示例（React/Vue 伪代码）：**
```html
<!-- 错误示范：所有行的删除按钮 testid 都一样 -->
<button data-testid="contact-btn-delete">删除</button>

<!-- 正确示范：结合数据 ID，确保每一行的按钮唯一 -->
<button data-testid={`contact-btn-delete-${item.id}`}>删除</button>
```

## 5. 落地流程与职责划分

为了确保规范的有效执行，将其融入日常的研发流程中：

### 5.1 产品经理职责（需求阶段）
在编写 PRD（需求文档）时，产品经理需在文档末尾附上《关键测试元素清单》，明确指出本次需求中哪些元素必须添加 `data-testid`，并给出推荐的命名。

**PRD 模板示例：**
> **测试标识要求（前端必填）**
> 本需求涉及以下关键交互，请前端在对应元素上添加 `data-testid`：
> - 保存漏斗按钮：`funnel-btn-save`
> - 添加步骤按钮：`funnel-btn-add-step`
> - 漏斗图表区域：`funnel-container-chart`

### 5.2 前端开发职责（开发阶段）
前端开发在实现 UI 时，严格按照 PRD 中的清单，在对应的 HTML 标签或组件上添加 `data-testid` 属性。此项工作将作为前端提测的**硬性验收标准**之一。

### 5.3 测试/产品职责（验收阶段）
在编写自动化测试脚本或进行功能验收时，直接使用 `[data-testid="xxx"]` 作为选择器进行元素定位。若发现遗漏，可打回要求前端补充。

## 6. 示例：漏斗分析 V2 改造清单

以漏斗分析 V2 为例，前端需在以下元素补充 `data-testid`：

| 页面区域 | 元素描述 | 推荐 data-testid |
|---|---|---|
| **顶部操作区** | 保存按钮 | `funnel-btn-save` |
| **左侧配置区** | 选择对象下拉框 | `funnel-select-object` |
| | 添加属性按钮 | `funnel-btn-add-step` |
| | 步骤项容器（动态） | `funnel-item-step-${index}` |
| | 步骤删除按钮（动态） | `funnel-btn-delete-step-${index}` |
| **数据筛选区** | 数据筛选 Tab | `funnel-tab-filter` |
| | 日期范围选择器 | `funnel-input-date-range` |
| **图表展示区** | 图表类型切换图标 | `funnel-icon-chart-type` |
| | 图表 Canvas 容器 | `funnel-container-chart` |
| **保存弹窗** | 报表名称输入框 | `funnel-input-report-name` |
| | 弹窗确认按钮 | `funnel-btn-modal-confirm` |
| | 弹窗取消按钮 | `funnel-btn-modal-cancel` |
